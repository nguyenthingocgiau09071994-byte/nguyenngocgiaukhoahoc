# Business Logic Audit

Scope: currently opened project only (`index.html`, `admin.html`, `app.js`, `admin.js`, `admin-content-library.js`, `firebase-config.js`, `firestore.rules`, Firebase config files, and visible product flows).

Verification performed: static product/QA review, Firebase rules/config review, script syntax checks for `app.js`, `admin.js`, `firebase-config.js`, and `admin-content-library.js`.

| Feature | Problem | Severity | Recommendation | Priority |
|---|---|---:|---|---:|
| Firebase security rules | Firestore rules allow public read/write on `content`, `members`, `users`, `qa`, `progress`, `settings`, `access`, `favorites`, and `notes`. This directly contradicts `security_spec.md` and exposes student data, roles, content, progress, and access plans. | Critical | Replace public rules with owner/admin rules: content writes admin-only, users self/admin only, progress/favorites/notes owner-only, access/settings admin-only, Q&A validated by signed-in identity. Add emulator tests for all Dirty Dozen cases. | P0 |
| Admin authorization | `admin.html` is directly accessible and admin capabilities are controlled client-side through localStorage role values. A user can self-edit localStorage or Firestore to become admin because rules permit writes. | Critical | Require Firebase Auth and server-enforced admin custom claim or protected `/users/{email}.role` checked by secure rules. Redirect non-admin users before rendering admin data. | P0 |
| Built-in admin credentials | `index.html` seeds `admin@masterclass.vn` with password `Admin@2026`, and `firebase-config.js` accepts that password as a bypass path. This is a production account takeover risk. | Critical | Remove hardcoded admin password and local seeding. Create admin via Firebase Console/custom claims only. Rotate credentials immediately. | P0 |
| Password storage | Local accounts store base64-encoded passwords in `academy_users`; "remember password" stores raw password in localStorage. This is not secure hashing and can be read by any script on the page. | Critical | Use Firebase Auth exclusively. Never store raw or reversible passwords in localStorage. Replace remember-password with Firebase session persistence. | P0 |
| User permissions and package access | Course/package access checks are performed in client-side JavaScript using `academy_access`. Users can change their own package in localStorage or Firestore due to open rules. | Critical | Enforce package access in Firestore rules and/or backend callable functions. Treat client checks as UI hints only. | P0 |
| Member data privacy | Admin member tables and Firebase listeners read all `members`, `users`, and `access` documents on any client where Firebase is enabled. Rules allow anyone to read them. | Critical | Restrict member/user/access collection reads to admin. Students should only read their own profile and access plan. | P0 |
| Firebase data deletion | Syncing `academy_content` deletes Firestore `content` docs not present in localStorage. A stale browser, empty localStorage, or malicious client can erase content. | Critical | Remove client-side bulk deletion. Use explicit admin-only delete action with confirmation and server-side authorization. | P0 |
| Q&A identity and integrity | Q&A is stored as arrays under `academy_qa_*` and synced as whole documents. Anyone can overwrite the entire questions array, spoof name/email, delete questions, or edit replies. | High | Store Q&A as subcollection documents with author UID/email, immutable author fields, size limits, and admin-only reply fields. | P0 |
| Firebase integration | App has dual sources of truth: Firebase Auth/Firestore and localStorage. Several flows write with `originalSetItem` and avoid sync, while others override `localStorage.setItem`. Data can diverge. | High | Define one source of truth. Use Firestore for persistent data and localStorage only for cache/session UI state. Add clear offline reconciliation rules. | P1 |
| Admin content library | `admin-content-library.js` exists and contains unified content management, but current `admin.html` only loads `admin.js`; the content library is not included. The intended "all posts in admin" feature may not run. | High | Add the content library script back or merge its functionality into `admin.js`. Verify admin content list contains all homepage/course/video content. | P0 |
| Content placement | Admin supports placement fields (`homepage`, `featured`, `video`, `curriculum`), but learner rendering mainly displays managed content in `publishedGrid`; curriculum placement is not fully integrated into course modules. | High | Map each placement to a real destination: homepage carousel/cards, video library, featured area, curriculum module rows. Add preview text "will appear in..." before save. | P1 |
| Draft/publish logic | Draft status is filtered in several learner renderers, but multiple render overrides exist. Because `app.js` has repeated render implementations, future edits can accidentally show drafts. | Medium | Consolidate content rendering to one function and add tests: draft hidden, published visible, access-gated hidden/visible by plan. | P1 |
| Admin save flow | Multiple scripts assign `contentForm.onsubmit` (`admin.js` base, routing, upload, final controller, content library). Last-loaded handler wins, making behavior fragile. | High | Use one editor controller. Remove duplicate submit handlers or dispatch named events. Document data contract for content items. | P1 |
| Data migration/default content | `admin.js`, `app.js`, and `admin-content-library.js` all seed/sync default content into `academy_content`. This can duplicate or overwrite fields and confuse admin counts. | Medium | Centralize default content seeding in one migration function with versioning and idempotency. | P2 |
| Error handling | Firebase sync catches mostly log to console; user sees no offline/failed-sync state. Some registration/member writes succeed locally even if Firebase fails. | High | Add visible sync status, retry queue, and "saved locally, not synced" messaging. Block business-critical success states until required backend writes succeed. | P1 |
| Loading states | Firestore `onSnapshot`, auth state, content loading, member loading, and admin lists have no explicit loading states; users may see stale local defaults before remote data arrives. | Medium | Add loading skeletons/spinners and "last synced" indicators for content, profile, admin lists, and Q&A. | P2 |
| Empty states | Some empty states exist (`adminEmpty`, `publishedEmpty`, Q&A empty), but not all are actionable. Empty admin/content states do not consistently guide the next action. | Low | Make every empty state actionable: "Add content", "Clear filters", "Ask first question", "No members yet". | P3 |
| Success states | Many actions use transient toast only. Important actions like registration, save content, permissions, avatar upload, and Q&A reply need persistent confirmation or visible updated row state. | Medium | Add inline success states after saves and optimistic row updates with rollback on failed sync. | P2 |
| Account registration | Local registration creates account and then opens login, but if Firebase create fails, user still sees success. Duplicate emails can overwrite local profiles. | High | Validate duplicate emails before write. Treat Firebase Auth creation as required when online. Show precise duplicate/weak-password errors. | P1 |
| Login flow | Login allows `Admin@2026` fallback for users and stores local session independently of Firebase Auth. This can create mismatched session, role, and Firestore identity. | High | Remove fallback password logic. Use Firebase Auth result as the only session source. Load role/access after auth. | P0 |
| Forgot password | Local fallback forgot-password updates local password directly, while Firebase path sends reset email. Depending on script load/state, user can have two password realities. | High | Use Firebase password reset only. Remove local password mutation. | P1 |
| Logout/session | `localStorage.removeItem('academy_session')` signs out Firebase only if current user exists. Other sensitive local data (`academy_users`, access cache, notes) remains. | Medium | On logout clear user-scoped cache or namespace it by user and stop displaying previous user's data. | P2 |
| Progress tracking | Progress is stored in `academy_progress_{email}` and can be edited by the client. Completion buttons and certificates rely on client data. | High | Store progress server-side with owner-only writes and validate lesson IDs. Certificate eligibility should be derived from trusted progress/quiz results. | P1 |
| Quiz and certificate | Quiz answers are static in HTML/client JS and certificate generation can be unlocked client-side by manipulating localStorage or DOM. | Medium | Treat quiz as practice unless server-validated. For real certificates, store attempts and eligibility in Firestore with validation. | P2 |
| Notes | Notes sync uses a single key `masterclass-note` plus user session email. Switching users can show stale notes before auth loads. | Medium | Store notes by user ID/email locally and remotely. Clear note UI while auth/loading state is unresolved. | P2 |
| Favorites/saved lessons | Favorites use shared `academy_favorites` key and are synced by current session email. Multi-user browser sessions can leak saved content between users. | Medium | Namespace favorites by authenticated user and render only after session resolution. | P2 |
| Avatar upload | Avatar is stored as base64 in localStorage/users collection. Large avatars can exceed localStorage/Firestore document limits and slow profile rendering. | Medium | Use Firebase Storage with file size/type validation, thumbnail generation, and URL stored in user profile. | P2 |
| Media upload/content images | Admin image upload stores compressed data URLs in localStorage/Firestore documents. This risks document bloat and failed sync. | Medium | Store images in Firebase Storage or asset hosting; save only URL and metadata in Firestore. | P2 |
| Video links | YouTube URL parsing supports common formats, but invalid links may save and only show weak preview text. | Low | Validate video URL before save. Show exact error and prevent publish if type is Video and no valid media exists. | P3 |
| Search/filter | Learner search appears tied to lessons, while admin search filters content title/summary. No empty recovery or filter summary is consistently shown. | Low | Add visible filter chips, "clear search", and count of results. | P3 |
| Navigation | Sidebar anchors and admin panel navigation mostly exist, but admin panel state is client-only and defaults dashboard on load. Deep-linking directly to a panel is not supported. | Low | Support query/hash routes such as `admin.html#content` and preserve last panel. | P3 |
| Mobile navigation | Mobile sidebar has controls and responsive CSS, but feature density is high and some workflows (admin editor, tables, profile modal) may overflow. | Medium | QA mobile breakpoints for admin editor, member table, content viewer, and lesson modal. Add stacked table/card layouts. | P2 |
| API calls | No custom backend API calls are present. Firebase SDK is the effective API layer, but it is called directly from the browser for privileged operations. | High | Move privileged operations to secure Firestore rules or Cloud Functions. Document which operations are client-safe. | P1 |
| Environment configuration | Real Firebase config is committed in `firebase-config.js` and `firebase-applet-config.json`. API key is not secret by itself, but public config plus open rules makes abuse easy. | High | Keep public config acceptable, but lock rules immediately. Use `.env`/build-time config for project switching and avoid duplicate config files with stale values. | P1 |
| Encoding/content quality | Several file reads show mojibake in comments and literals. If users see this in browser or generated UI, Vietnamese copy and icons may appear broken. | Medium | Normalize all source files to UTF-8 without mojibake, then add a smoke test that checks key Vietnamese strings render correctly. | P2 |
| Build/runtime split | `src/App.tsx` renders an empty div, while the actual product is static `index.html`/`app.js`. Vite build may not represent the live product. | Medium | Decide whether the product is static or React. Align build/deploy pipeline with the actual app entrypoint and remove unused scaffold if not needed. | P2 |
| Deployment correctness | `index.html` references `styles.css?v=20260708-0200` while `app.js` remains `v=20260708-0180`; admin uses `0180`. Cache busting is manual and inconsistent. | Low | Use a build process or single version constant for all assets. Avoid manual query strings. | P3 |
| Business package model | Membership sales UI exists in `index.html`, but some scripts hide membership sales. Admin permission model includes 699K/999K/Mentoring while learner plan cards show Starter/Professional/Mentoring. | Medium | Finalize package taxonomy and map product labels, prices, and access codes consistently across learner, admin, and Firestore. | P1 |
| Dashboard metrics | Many metrics are hardcoded (`68%`, `12 days`, `5 chapters · 28 lessons`, completion counts) and may not match user data. | Medium | Calculate dashboard metrics from actual progress/content data and show loading/empty states when not available. | P2 |
| Notifications | Notifications are mostly local/generated from localStorage, with badge counts based on client state. No read/unread persistence per authenticated admin/user in backend. | Low | Persist notification read state per user/admin in Firestore and derive counts from trusted data. | P3 |
| Admin CSV export | CSV export includes member emails/phones and is available client-side wherever admin UI is accessible. With current permissions, this is a privacy leak. | High | Gate export behind verified admin auth and log/export consent. Restrict member reads first. | P0 |
| Validation | Forms rely mainly on HTML required/minlength and light JS checks. Phone format, duplicate content IDs, package values, role values, Q&A length, and content size are not consistently validated. | Medium | Add schema validation before local write and Firestore rules validation before remote write. | P2 |
| Audit/test coverage | No automated product tests or Firestore rules tests are present despite `security_spec.md` describing them. | High | Add emulator tests for rules, Playwright smoke tests for login/register/admin/content/Q&A, and data contract tests for localStorage/Firestore sync. | P1 |

## Priority Summary

- P0: Fix security and permission enforcement before any real student/customer data is used.
- P1: Consolidate content/admin/auth data flows so features behave predictably.
- P2: Improve reliability, validation, loading states, and data ownership.
- P3: Polish navigation, empty states, filters, and cache/deploy ergonomics.
